# Agoralo

**The modern agora for communities that act together.**

Agoralo is a modular community platform for associations, alumni groups, civic networks, local communities, organizations, learning groups, professional networks and purpose-driven communities.

It is not another social media feed.

Agoralo is designed as a **modern digital agora**: a place where people gather, deliberate, exchange, learn, vote, organize and act together.

---

## Vision

The internet gave us social media, but social media gradually turned communities into audiences.

Facebook, Instagram, LinkedIn, MySpace and similar platforms created enormous networks, but over time many of them became media platforms: optimized for feeds, attention, advertising, influencers and content consumption.

Agoralo aims to recover the older and deeper idea of the **Agora** and the **Forum**:

> A trusted public or private space where people meet, discuss, exchange, decide, coordinate and act.

Ancient communities had the **Agora**.
Roman society had the **Forum**.
Modern communities need a programmable, web-scale version of both.

Agoralo is that foundation.

Agoralo is based on a simple architectural principle:

> The platform should provide the shared community layer. Vertical modules should provide the specific things each community wants to do.

The shared community layer includes:

* Roles
* Permissions

## Development Principle

The most important technical rule:

> Do not rewrite HumHub first. Do not heavily modify HumHub core unless absolutely necessary.

Preferred extension order:

1. Configuration
2. Themes
3. HumHub modules
4. REST API integrations
5. External services
6. Core modifications only if unavoidable

This keeps the project upgrade-safe and reduces long-term maintenance risk.

## Hosting Model

Agoralo should initially use a self-hosted model.

Recommended architecture:

```text
agoralo.com          → public marketing website
app.agoralo.com      → main platform instance
demo.agoralo.com     → public demo/sandbox
client1.agoralo.com  → future client/community instance
```

Early production model:

```text
One community = one HumHub installation
One installation = one database
One installation = one upload directory
One installation = one backup policy
```

Do not build full multi-tenancy at the beginning.

Recommended server for MVP:

```text
Debian 12
Nginx
PHP-FPM
MariaDB
Let’s Encrypt SSL
Off-server backups
SMTP provider
2–4 vCPU
4–8 GB RAM
80–160 GB SSD
```

For the first serious MVP, a comfortable starting point is:

```text
4 vCPU
8 GB RAM
160 GB SSD
```

## Security and Privacy Principles

Agoralo will handle sensitive community data.

Security and privacy must be foundational.

Principles:

* No hardcoded secrets
* Use environment variables
* Use HTTPS everywhere
* Use off-server backups
* Respect role-based permissions
* Preserve privacy controls
* Avoid unnecessary personal data collection
* Keep audit logs for sensitive actions
* Support data export where practical
* Design with GDPR in mind
* Do not send private community data to AI services without explicit configuration and consent
